## 에이전트란? (교재 p.258)

### AI 에이전트의 개념

**에이전트(Agent)** 는 '대리인'이라는 뜻으로, 주어진 목표와 규칙에 따라 스스로 판단하고 작업을 수행하는 존재입니다.

### AI 서브에이전트

- 클로드 코드에서 특정 작업을 전문적으로 수행하는 독립형 AI 도우미
- 각각 고유한 역할, 권한, 전문 분야를 가짐
- 메인 AI(클로드 코드)가 서브에이전트에게 작업을 위임

### 서브에이전트 vs 커스텀 명령어

| 구분 | 커스텀 명령어 | 서브에이전트 |
|------|-------------|-------------|
| 성격 | 정해진 규칙에 따라 실행 | 상황을 판단하여 유연하게 대응 |
| 비유 | 레시피대로 따라하는 요리사 | 재료를 보고 창의적으로 요리하는 셰프 |
| 적합한 작업 | 반복적·정형화된 작업 | 복잡한 판단이 필요한 작업 |

## 서브에이전트 협업 (교재 p.273)

### 3개 에이전트 동시 활용

- 코드 리뷰어가 버그와 보안 이슈 발견
- 최적화 에이전트가 성능 개선 포인트 제안
- UX 에이전트가 사용성 개선 사항 도출

### 직렬 처리 vs 병렬 처리 (교재 p.274)

| 처리 방식 | 설명 | 적합한 경우 |
|-----------|------|-------------|
| 직렬 처리 | 순서대로 하나씩 실행 | 앞 작업 결과가 뒤 작업에 필요할 때 |
| 병렬 처리 | 여러 작업을 동시에 실행 | 작업 간 의존성이 없을 때 |

> 클로드 코드가 작업 성격에 따라 직렬/병렬 방식을 자동으로 선택합니다. 코드 리뷰 3개를 동시에 요청하면 병렬로, 리뷰 후 수정은 직렬로 진행됩니다.

## 작업 결과 백업 (교재 p.280)

### 백업 명령어
현재 상태를 백업해 줘

- 클로드 코드가 `backup` 디렉터리를 자동 생성하고 현재 파일 상태를 복사
- 서브에이전트가 코드를 수정하기 전에 백업하면 안전

> 💡 서브에이전트가 코드를 대량으로 수정하기 전에 반드시 백업하세요! 백업 명령 한 줄이면 현재 상태를 안전하게 보존할 수 있습니다.

## 셀프 체크 — 07-1 (교재 p.281)

**1.** 커스텀 명령어와 에이전트의 기술적 차이점으로 옳은 것을 고르세요.

- ① 커스텀 명령어는 `.md` 파일로 저장되지만 에이전트는 `.json` 파일로 저장된다.
- ② **에이전트는 독립적인 AI 모델을 선택할 수 있지만 커스텀 명령어는 메인 세션의 모델을 공유한다.** ✅
- ③ 커스텀 명령어가 에이전트보다 더 복잡한 작업 플로우를 처리할 수 있다.
- ④ 에이전트는 적용을 위해 재시작이 필요하지만 커스텀 명령어는 바로 사용할 수 있다.

**2.** 에이전트 생성 시 '4. Inherit from parent' 옵션을 선택했을 때의 동작으로 옳은 것을 고르세요.

- ① 항상 가장 저렴한 모델을 자동 선택한다.
- ② **메인 세션에서 사용 중인 모델을 그대로 상속받는다.** ✅
- ③ 부모 에이전트가 사용하는 모델을 복사한다.
- ④ Claude의 최신 모델로 자동 업그레이드된다.

**3.** 에이전트에게 Read-only 권한만 부여했을 때 수행할 수 없는 작업을 모두 고르세요.

- ① 코드 파일 내용 읽기 및 분석
- ② 버그 리포트 생성
- ③ **발견한 버그 수정** ✗ (불가)
- ④ **새로운 테스트 파일 생성** ✗ (불가)

## 5W1H 프레임워크 (교재 p.289)

### 제품 요구사항 정리

서브에이전트 팀에게 작업을 지시하기 전에 5W1H 프레임워크로 요구사항을 명확히 합니다.

| 항목 | 질문 | 예시 (AI 공감 다이어리) |
|------|------|------------------------|
| **Who** | 누가 구현하나? | 백엔드·프런트엔드·QA 에이전트 |
| **What** | 무엇을 만드나? | 일기 작성 → AI가 감정 분석 → 공감 답변 |
| **When** | 언제 만드나? | 지금 바로 개발 시작, 즉시 완성 |
| **Where** | 어디서 사용하나? | 웹 브라우저 |
| **Why** | 왜 필요한가? | 혼자 감정 정리가 어려운 사람에게 위로 |
| **How** | 어떻게 구현하나? | OpenRouter API + 감정 분석 AI |

> 💡 5W1H 프레임워크로 요구사항을 명확히 정의한 뒤 역할별 에이전트에게 작업을 지시하면, 실제 팀 개발과 동일한 협업이 가능합니다.

## 에이전트 팀 협업 사례 (교재 p.289)

### 사례 1: AI 공감 다이어리

1. 사용자 → 5W1H로 요구사항 정의
2. 백엔드 → OpenRouter API로 감정 분석 로직 구현
3. 프런트엔드 → 따뜻한 일기장 느낌의 UI 구현
4. QA → 기능 테스트 수행

### 사례 2: PDF 요약 AI (교재 p.294)

1. PM → 5W1H 기반 요구사항 정의
2. 백엔드 → PDF 텍스트 추출 로직
3. AI 통합 → 요약 및 핵심 키워드 추출 AI 연동
4. 프런트엔드 → PDF 업로드 → 요약 결과 표시 UI
5. QA → 전체 기능 테스트

> 5개 서브에이전트를 총동원하면 역할별 전문성을 살려 빠르고 체계적인 개발이 가능합니다.

## 셀프 체크 — 07-2 (교재 p.300)

**1.** AI 공감 다이어리 개발에서 QA 테스트 시간이 더 오래 걸린 이유로 가장 적절한 것을 고르세요.

- ① QA 에이전트가 더 느린 AI 모델을 사용하기 때문이다.
- ② **QA 에이전트는 이전 에이전트들의 작업 결과를 모두 검토해야 하기 때문이다.** ✅
- ③ QA 에이전트만 인터넷 연결이 필요하기 때문이다.
- ④ QA 에이전트는 파일 읽기 권한이 없기 때문이다.

**2.** PM 에이전트가 생성한 PRD를 다른 에이전트들이 참조하는 방식으로 옳지 않은 것을 고르세요.

- ① PRD가 프로젝트 폴더에 생성되어 각 에이전트들이 접근한다.
- ② 에이전트들이 순차적, 혹은 병렬로 작업하며 PRD를 참조하게 된다.
- ③ PRD 파일 경로를 에이전트별로 지정하지 않아도 된다.
- ④ **PRD는 `.claude/agents/` 폴더에 자동으로 복사되어 공유된다.** ✗ (옳지 않음)

## MCP란? (교재 p.304)

### MCP(Model Context Protocol)

**MCP**는 AI 모델이 외부 데이터, 도구, 서비스와 연결할 때 사용하는 **표준화된 통신 규약**입니다.

### API vs MCP (교재 p.305)

| 구분 | API | MCP |
|------|-----|-----|
| 비유 | 특정 가게 하나의 대화 창구 | 여러 가게를 한꺼번에 관리하는 표준 규격 |
| 연결 대상 | 특정 서비스 1개 | 다양한 데이터·도구를 통합 관리 |
| 사용자 | 개발자가 직접 코드 작성 | AI가 스스로 도구를 선택하여 활용 |

### 클라이언트-서버 모델 (교재 p.306)

- **클로드 코드** = MCP 클라이언트 (요청하는 쪽)
- **노션, 깃허브, Playwright 등** = MCP 서버 (서비스를 제공하는 쪽)

> 💡 MCP 덕분에 클로드 코드가 노션에 문서를 쓰고, 깃허브에 코드를 올리고, 브라우저를 자동 제어할 수 있습니다!

## 로컬 MCP vs 원격 MCP (교재 p.315)

### 두 가지 MCP 유형

| 구분 | 로컬 MCP | 원격 MCP |
|------|---------|---------|
| 실행 위치 | 내 컴퓨터 안에서 직접 실행 | 인터넷상 외부 서버에서 실행 |
| 인터넷 | 필요 없음 | 반드시 필요 |
| 설정 | 조금 더 복잡 | 간단 (URL만 지정) |
| 속도 | 빠름 (로컬 실행) | 네트워크에 따라 다름 |
| 보안 | 개인정보 보호에 유리 | 인증 절차 필요 |

### 원격 MCP 명령어 구조

## MCP란? (교재 p.304)

### MCP(Model Context Protocol)

**MCP**는 AI 모델이 외부 데이터, 도구, 서비스와 연결할 때 사용하는 **표준화된 통신 규약**입니다.

### API vs MCP (교재 p.305)

| 구분 | API | MCP |
|------|-----|-----|
| 비유 | 특정 가게 하나의 대화 창구 | 여러 가게를 한꺼번에 관리하는 표준 규격 |
| 연결 대상 | 특정 서비스 1개 | 다양한 데이터·도구를 통합 관리 |
| 사용자 | 개발자가 직접 코드 작성 | AI가 스스로 도구를 선택하여 활용 |

### 클라이언트-서버 모델 (교재 p.306)

- **클로드 코드** = MCP 클라이언트 (요청하는 쪽)
- **노션, 깃허브, Playwright 등** = MCP 서버 (서비스를 제공하는 쪽)

> 💡 MCP 덕분에 클로드 코드가 노션에 문서를 쓰고, 깃허브에 코드를 올리고, 브라우저를 자동 제어할 수 있습니다!

## 로컬 MCP vs 원격 MCP (교재 p.315)

### 두 가지 MCP 유형

| 구분 | 로컬 MCP | 원격 MCP |
|------|---------|---------|
| 실행 위치 | 내 컴퓨터 안에서 직접 실행 | 인터넷상 외부 서버에서 실행 |
| 인터넷 | 필요 없음 | 반드시 필요 |
| 설정 | 조금 더 복잡 | 간단 (URL만 지정) |
| 속도 | 빠름 (로컬 실행) | 네트워크에 따라 다름 |
| 보안 | 개인정보 보호에 유리 | 인증 절차 필요 |

### 원격 MCP 명령어 구조

```bash
claude mcp add --transport http [이름] [서버 URL]
```

### 로컬 MCP 명령어 구조

```bash
claude mcp add [이름] [옵션] -- [MCP 실행 명령]
```

## MCP 설치 명령어 분석 (교재 p.316)

### Sequential Thinking MCP 설치 명령어

```bash
claude mcp add sequential-thinking -s local \
  -- cmd /c npx -y \
  @modelcontextprotocol/server-sequential-thinking@latest
```

### 명령어 구성 분석

| 부분 | 의미 |
|------|------|
| `sequential-thinking` | 클로드 코드에서 이 MCP를 구분하기 위한 이름 |
| `-s local` | 현재 프로젝트에만 적용 (scope 설정) |
| `--` | 클로드 코드 옵션과 MCP 실행 명령의 구분자 |
| `cmd /c` | 윈도우 환경에서 명령 실행 (맥/리눅스에서는 생략) |
| `npx -y` | Node.js 패키지를 자동 설치하여 실행 |
| `@model...@latest` | 패키지명@버전 (최신 버전) |

> ⚠️ 윈도우 파워셸 환경에서는 `cmd /c` 부분을 삭제합니다!

## 인증이란? (교재 p.316)

### 왜 인증이 필요한가?

외부 서비스와 MCP를 연결할 때, 접속하려는 사람이 정말 허가된 사용자인지 확인하기 위해 **인증(Authentication)** 이 필요합니다.

### 인증 방식

- **원격 MCP** → 인터넷을 통해 통신하므로 인증 절차 필수
- **로컬 MCP** → 내 컴퓨터에서 직접 실행하므로 인증 불필요

> 원격 MCP는 API 키, OAuth 등 인증이 필수이고, 로컬 MCP는 내 컴퓨터에서 실행하므로 별도 인증 없이 바로 사용할 수 있습니다.

### 왜 버전 관리가 필요한가?

| ② 커밋(commit) | `git commit` | 현재 상태를 로컬 저장소에 저장 |
| ③ 푸시(push) | `git push` | 로컬 → 원격 저장소(깃허브)에 업로드 |

## 실습 ③ 노션 MCP 연결 (교재 p.306)

### 노션 MCP 설치

```bash
claude mcp add --transport http notion https://mcp.notion.com/mcp
```

### 연결 확인 순서

1. 설치 후 브라우저에서 노션 로그인 및 인증(Authorize) 완료
2. 클로드 코드에서 `/mcp` 입력 → `notion: connected` 확인
3. View tools → 15개 도구 확인 (`notion-search`, `notion-fetch` 등)

### 노션 MCP로 리서치 노트 만들기 (교재 p.313)

클로드 코드의 업데이트, 변경 사항을 검색하고 요약해 줘.
그 결과를 노션 MCP를 이용해 노션에 저장해 줘.

### ✅ 확인 체크리스트

## 실습 ④ (계속) — 순차적 추론 성능 비교 (교재 p.318)

### 순차적 추론 vs 일반 추론 비교 프롬프트

내 웹 포트폴리오 방문자의 체류 시간을 지금의 2배로 늘리고 싶어.
두 가지 문서를 작성해.

노션 MCP를 이용해 '체류 시간 늘리기'로 저장해 줘.
Sequential Thinking MCP를 이용해서 체계적으로 구상하고
'체류 시간 늘리기-체계적 구상'으로 저장해 줘.

### 비교 결과

| 방식 | 결과 분량 |
|------|----------|
| 일반 추론 | 총 85줄 |
| 순차적 추론 | 총 222줄 (약 2.6배 상세) |

> 💡 Sequential Thinking은 복잡한 문제를 여러 단계로 나눠 사고하므로 일반 추론보다 더 체계적이고 상세한 결과를 생성합니다.

## 실습 중 자주 발생하는 문제

| 문제 | 원인 | 해결 방법 |
|------|------|----------|
| MCP 설치 후 connected 안 됨 | Node.js 미설치 또는 버전 오래됨 | `node -v` 확인, v18 이상 필요 |
| 노션 인증 실패 | 브라우저 팝업 차단 | 팝업 허용 후 재시도 |
| `cmd /c` 에러 (파워셸) | 파워셸에서 `cmd /c` 불필요 | `cmd /c` 부분 제거 |
| Sequential Thinking 응답 없음 | npx 패키지 다운로드 지연 | 네트워크 확인 후 재시도 |
| Playwright 브라우저 안 열림 | Chromium 미설치 | `npx playwright install chromium` 실행 |
| 깃허브 MCP 인증 실패 | API 토큰 만료 또는 권한 부족 | 토큰 재발급 (`repo`, `user`, `project` 권한) |

## 핵심 키워드 — 이론 07-1 (교재 p.281)

### 07-1: 6가지 키워드

- **에이전트** — 사람이 직접 지시하지 않아도 목표와 규칙에 따라 스스로 판단하고 작업을 수행하는 AI 대리 프로그램
- **AI 서브에이전트** — 클로드 코드에서 특정 작업을 전문적으로 수행하도록 만든 독립형 AI 도우미
- **/agents** — 클로드 코드의 전용 명령어, 새로운 서브에이전트를 생성하거나 관리
- **코드 리뷰** — 코드의 버그와 규칙 위반을 점검하고 개선점을 제안하는 품질 검토 과정
- **최적화** — 프로그램의 속도, 자원 사용, 효율성을 향상시키는 성능 개선 작업
- **UX 디자인** — 사용자가 쉽고 편리하게 이용할 수 있도록 화면 구성과 인터페이스를 설계하는 과정

## 핵심 키워드 — 이론 07-2 (교재 p.300)

### 07-2: 5가지 키워드

- **제품 기획 관리자** — 제품의 목표, 기능, 사용자 요구사항을 정의하고 전체 개발 일정을 관리
- **백엔드 개발자** — 서버, 데이터베이스 설계와 API 연동 등 핵심 로직을 구현하는 기술 담당자
- **프런트엔드 개발자** — 사용자가 보는 화면과 인터페이스를 설계하고 구현하는 UI 개발자
- **품질 보증 엔지니어** — 기능과 성능을 테스트해 버그를 수정하고 서비스의 안정성을 확보
- **AI 통합 전문가** — AI 모델을 연동, 최적화하기 위해 기술 통합을 담당

## 핵심 키워드 — 실습 08-1 (교재 p.321)

### 08-1: 6가지 키워드

- **MCP** — AI 모델이 외부 데이터, 도구, 서비스와 연결할 때 사용하는 표준화된 통신 규약
- **클라이언트-서버 모델** — 클로드 코드(클라이언트)가 MCP 서버에 요청하고 응답받는 구조
- **로컬 MCP** — MCP 중개 프로그램을 내 컴퓨터 안에서 직접 실행하는 방식
- **원격 MCP** — 그 중개 프로그램이 인터넷상 외부 서버에서 실행하는 방식
- **인증** — 접속하려는 사람이 정말 허가된 사용자인지 확인하는 절차
- **이중 하이픈 `--`** — 두 글자 이상인 옵션 이름을 표시하거나, 명령어와 옵션을 구분할 때 쓰이는 기호

## 핵심 키워드 — 실습 08-2 (교재 p.345)

### 08-2: 7가지 키워드

- **테스트** — 코드가 의도한 대로 실행되는지, 오류가 없는지 확인하는 과정
- **버전 관리** — 코드의 변경 이력을 기록하고, 필요할 때 이전 상태로 되돌릴 수 있게 함
- **Context7** — 클로드 코드가 항상 최신 문서를 참고하도록 돕는 지식 연결 도구
- **Playwright** — 웹 사이트를 자동으로 테스트하고 UI를 검증하는 브라우저 제어 도구
- **추가(add)** — 깃허브에 업로드할 파일들을 선택해서 준비 공간에 추가
- **커밋(commit)** — 준비된 파일을 로컬 저장소에 저장
- **푸시(push)** — 로컬 저장소에 커밋된 파일을 원격 저장소로 업로드

## 핵심 키워드 — 실습 08-3 (교재 p.368)

### 08-3: 6가지 키워드

- **배포** — 내 애플리케이션을 외부 인터넷에 연결된 서버에 올리는 과정
