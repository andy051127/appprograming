## 프롬프트의 중요성 [cite: 232-239]
* AI는 방대한 언어 데이터를 학습한 거대한 지식 저장소입니다.
* 이 거대한 지식 창고와 소통할 수 있는 통로는 **프롬프트 창 하나**뿐입니다.
* 프롬프트는 내비게이션에 목적지를 입력하는 것과 비슷합니다.
    * "좋은 데 데려다 줘" (명확하지 않음) → 정확한 안내 불가
    * "서울역에서 강남역까지 가장 빠른 길" (명확함) → 효율적인 루트 도달
* 제한된 언어로 AI의 잠재력을 최대한 끌어내기 위해서는 **명확하고 구체적인 프롬프트 작성**이 필수적입니다.

## 좋은 프롬프트의 3가지 조건 [cite: 240-244]
| 조건 | 설명 | 예시 |
| :--- | :--- | :--- |
| **명확성** | 모호한 표현을 피하고, 원하는 결과를 구체적으로 전달 | "멋진" → "밝고 깔끔한 스타일" |
| **구체성** | 결과물에 필요한 기능이나 조건을 세부적으로 지시 | "시계와 검색바도 추가해 줘" |
| **맥락 제공** | 왜 필요한지, 어떤 상황에서 활용되는지 알려주기 | "취업용 마케팅 포트폴리오야" |

## 모호한 프롬프트의 한계 [cite: 245-254]
* 모호한 지시어는 AI의 판단을 흐리게 만들고, 기대와 다른 결과물을 초래합니다.

### 나쁜 프롬프트
> "멋진 나만의 시작 홈페이지를 만들어 줘."
* **문제점:** "멋지다"는 표현이 지나치게 모호하여 AI가 임의로 결정함.

### 개선된 프롬프트
> "오늘의 날씨와 주요 뉴스를 보여 주는 나만의 시작 홈페이지를 만들고 싶어. 밝고 깔끔한 스타일을 선호해. 시계와 검색바도 추가해 줘. 기술 관련 뉴스를 보여 줘."

* **Tip:** 처음부터 완벽하게 쓸 필요는 없습니다. AI가 질문을 되묻는 과정 자체가 프롬프트를 발전시키는 좋은 방법입니다!

## 5W1H: 체계적인 프롬프트의 핵심 틀 [cite: 255-260]
| 질문 | 의미 | 역할 |
| :--- | :--- | :--- |
| **Why** | 왜 만드는가? | 목적을 명확히 |
| **Who** | 누구를 위한 것인가? | 대상을 정의 |
| **What** | 무엇을 만들 것인가? | 기능을 구체화 |
| **When** | 언제 사용되는가? | 사용 맥락 파악 |
| **Where** | 어디서 사용되는가? | 사용 환경 정의 |
| **How** | 어떻게 만들 것인가? | 구현 방법 결정 |

* 소프트웨어 기능 명세인 **PRD 구조**에서는 특히 **What**과 **How**가 강조됩니다.

## PRD - 제품 기획서란? [cite: 261-266]
개발자들이 코딩에 앞서 작성하는 **제품 기획서(PRD: Product Requirements Document)**입니다.

| PRD | 비교 | 좋은 프롬프트 |
| :--- | :---: | :--- |
| 무엇을 만들지 명확히 정의 | **=** | 해야 할 일을 구체적으로 지시 |
| 프로젝트 목적과 배경 제시 | **=** | 왜 필요한지, 어떤 상황인지 설명 |
| 요구사항·구현 방법 구체화 | **=** | 필요한 기능과 조건 세부 명시 |

**좋은 프롬프트는 곧 잘 만들어진 기획서입니다!**

---

## 효과적인 프롬프트를 위한 기본 질문 템플릿 [cite: 267-278]
5W1H를 기반으로 어떤 프로젝트에나 적용할 수 있는 범용 템플릿입니다.

> **[무엇]**을 만들려고 해.
> 주요 기능은 **[구체적인 기능 설명]**이야.
> **[누구]**를 위한 것이고, **[왜/어떤 문제를 해결]**하기 위해 필요해.
> 어떻게 구현하면 좋을지 기술적인 방향과 함께 구체적인 PRD를 작성해 줘.

**핵심 포인트:**
* 5W1H의 모든 요소가 적절히 포함되어야 합니다.
* 구현 방향을 구체적으로 요구해야 합니다.
* AI를 위해 구체적인 기능을 기술적으로 설명해야 합니다.

## 4단계 포트폴리오 전략 [cite: 318-332]
큰 프로젝트를 4단계로 분해(컴퓨팅 사고!)하여 단계별로 AI에게 요청합니다.

[다이어그램: 각 단계별로 '요청→생성→확인→수정'이 반복되는 순환 구조]

| 단계 | 목표 | 핵심 활동 |
| :--- | :--- | :--- |
| **1. 뼈대** | HTML 기본 구조 | 섹션 구성, 고유 이름(ID) 부여 |
| **2. 기능** | 실제 데이터 | 경력, 성과, 케이스 스터디 |
| **3. 디자인** | 시각적 완성도 | 레퍼런스, 색상·레이아웃 |
| **4. 점검** | 품질 검증 | PRD 대조, 링크 확인 | [cite: 333]

---

## 왜 4단계로 나누는가? [cite: 335-345]
### 한 번에 모든 것을 요청하면?
* HTML, CSS, JavaScript가 뒤섞인 복잡한 코드가 한꺼번에 생성됨 [cite: 337]
* 모든 요청 사항이 반영되었는지 확인하기 어려움 [cite: 338]
* 이후 수정 과정에서도 불편함이 따름 [cite: 339]

### 단계별로 나누면? (컴퓨팅 사고의 '분해'!)
* 각 단계마다 **요청-생성-확인→수정**의 짧은 사이클 반복 (패턴 인식!) [cite: 341]
* 변경 위험을 줄이고, 구조와 품질을 차근차근 확보 [cite: 342]
* 점점 원하는 형태에 가까운 웹페이지를 완성 [cite: 343]

> **4단계 전략 = 컴퓨팅 사고의 '분해'와 '알고리즘'을 실전에 적용하는 것!** [cite: 344]

## 실습 ② 1단계: 뼈대 잡기 [cite: 358-359]

### 프롬프트 입력 [cite: 360-362]

### [확인] 체크리스트 [cite: 363-366]

- [ ] 기능별로 구분되어, 차후 특정 요소를 지목해 수정할 수 있는가?

> **핵심:** "기능은 아직 구현하지 말고, 구조만" — 단계를 분리하라고 명시하는 것이 핵심입니다[cite: 367].

"내용을 채워 줘"가 아니라 구체적이고 사실적인 정보를 제공해야 합니다[cite: 386].

### 프롬프트 입력 (경력 정보) [cite: 387-394]
> "나의 경력 포트폴리오는 다음과 같아. 섹션별로 다음 내용을 참고해서 업데이트해 줘.
> - 나는 3년차, 퍼포먼스/콘텐츠 마케팅 전문가야.
> - 핵심 성과: ROAS 200%, 25개 캠페인, 연 8억 매출 기여
> - 3년간 포지션 변화: 주니어(2023) → 퍼포먼스 마케터(2024) → 매니저+자문(2025)
> - 스킬 레벨: 퍼포먼스 마케팅 75%, 콘텐츠 마케팅 70%, 데이터 분석(GA4) 65%"

### 프롬프트 입력 (케이스 스터디) [cite: 395-401]
> "다음은 나의 대표 케이스 스터디 이력이야.
> - 이커머스: 월 광고비 1천만원, ROAS 200%
> - 스타트업: 인스타 팔로워 2천→8천(6개월) 증가
> - 로컬 브랜드: 블로그 방문자 500→3천(4개월) 증가"

## 핵심 키워드 정리
### 02-1 이론 키워드 [cite: 433-437]
* **프롬프트(Prompt):** AI와 소통하는 유일한 창구, AI 성능을 좌우하는 핵심
* **$5W1H$:** Why, Who, What, When, Where, How 체계적 사고의 틀
* **PRD:** 개발자들이 사용하는 제품 기획서 (Product Requirements Document)
* **템플릿:** 효과적인 프롬프트를 위한 재사용 가능한 질문 구조

### 02-2 실습 키워드 [cite: 439-445]
* **4단계 전략:** 뼈대 → 기능 → 디자인 → 점검 순서로 AI에게 요청
* **섹션(Section):** 웹페이지를 주제별로 구분한 구역 단위
* **뼈대:** 기본 HTML 구조만으로 틀을 잡는 단계
* **기능:** 실제 데이터를 채워 넣고 동작을 추가하는 단계
* **디자인:** 색상, 레이아웃, 시각적 요소를 적용하는 단계
* **점검:** 누락된 기능이나 오류를 수정하는 최종 단계
